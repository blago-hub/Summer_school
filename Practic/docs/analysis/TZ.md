# Техническое задание: Веб-приложение для скалодрома «Вертикаль»

---

### 1.1. Наименование проекта

Веб-приложение для клиентской записи на тренировки скалодрома «Вертикаль» (далее — Приложение).

## 2. Назначение и цели создания системы

### 2.1. Назначение системы

Приложение предназначено для автоматизации процесса записи клиентов скалодрома «Вертикаль» 
на групповые тренировки. Система заменяет ручную обработку заявок в Telegram и бумажном журнале, 
предоставляя клиентам возможность самостоятельно просматривать расписание, бронировать места, 
получать напоминания и оставлять отзывы об инструкторах.

### 2.2. Цели создания системы

**Бизнес-цели:**

- Снизить нагрузку на администратора при обработке записей (BR-001)
- Исключить двойные бронирования и путаницу в расписании (BR-001)
- Повысить возвращаемость клиентов через систему лояльности (BR-002)
- Получать обратную связь для оценки качества инструкторов (BR-003)

**Технические цели:**

- Интеграция с существующим бэкендом скалодрома через API (без дублирования данных)
- Адаптивный веб-интерфейс для мобильных устройств
- Надёжная обработка ошибок и сетевых сбоев

### 2.3. Критерии успеха проекта

1. **Конверсия записи:** >30% пользователей, открывших расписание, записываются на тренировку
2. **Время до записи:** <60 секунд от открытия приложения до подтверждённой брони
3. **Процент отмен:** <20% броней отменяются клиентами
4. **Процент оценок:** >40% клиентов оставляют оценку после тренировки
5. **Возвращаемость:** >50% клиентов записываются повторно в течение месяца

## 3. Характеристика объектов автоматизации

### 3.1. Описание предметной области

Скалодром «Вертикаль» — спортивный объект в бывшем складском ангаре, проводящий групповые тренировки по скалолазанию третий год.

**Ключевые параметры:**

- 5 штатных инструкторов + подработчики в сезон
- 2 формата тренировок: болдеринг (новички, до 8 человек) и трассы с верёвкой (опытные, до 16 человек)
- Длительность тренировки: 1.5 часа
- Расписание формируется на 7 дней вперёд
- Прокат снаряжения: скальники + страховочная система (фиксированная цена)
- Оплата: наличные или перевод на карту администратору (на месте)

### 3.2. Текущий процесс (AS-IS)

1. Клиент пишет в Telegram или звонит администратору
2. Администратор вручную проверяет расписание в тетрадке
3. Администратор подтверждает запись или отказывает
4. Клиент получает текстовое сообщение с деталями
5. Напоминания не отправляются (клиенты забывают)
6. После тренировки оценка инструктору не собирается системно

**Проблемы текущего процесса:**

- Двойные бронирования в часы пик
- Потеря заявок в потоке сообщений
- Отсутствие напоминаний → неявки клиентов
- Нет данных для оценки качества инструкторов
- Высокая нагрузка на администратора

### 3.3. Целевой процесс (TO-BE)

1. Клиент открывает веб-приложение на телефоне
2. Видит расписание на 7 дней вперёд с фильтрами
3. Выбирает слот, указывает потребность в прокате снаряжения
4. Подтверждает бронирование (мгновенно)
5. Получает email-подтверждение и напоминание за 2 часа
6. Посещает тренировку, оплачивает на месте
7. Оставляет оценку инструктору в течение 24 часов
8. После 10 посещений получает статус "Постоянный клиент" и скидку 10%

## 4. Требования к системе

### 4.1. Требования к системе в целом

#### 4.1.1. Архитектурные требования

- **Тип приложения:** Веб-приложение (Single Page Application)
- **Адаптивность:** Mobile-first, корректное отображение на экранах от 320px
- **Интеграция:** REST API с существующим бэкендом скалодрома (black-box)
- **Аутентификация:** JWT-токены после SMS-верификации
- **Хранение данных:** Локальная БД приложения для Client, Booking, Review, Notification; Read-only кэш для TrainingFormat, Instructor, TrainingSlot

#### 4.1.2. Разделение ответственности

| Категория | Сущности | Управление |
|-----------|----------|------------|
| Read-only | TrainingFormat, Instructor, TrainingSlot | Существующий бэкенд (внешний API) |
| Mutable | Client, Booking, Review, Notification | Наше приложение (локальная БД + API) |

#### 4.1.3. Ограничения

- **R-001:** Только клиентская роль (инструкторы и администраторы используют существующие интерфейсы)
- **R-002:** Веб-приложение (не нативное мобильное)
- **R-003:** Оплата на месте (без интеграции платёжных систем)
- **R-004:** Бэкенд скалодрома — black-box, гарантия атомарности бронирования на его стороне
- **R-005:** Срок разработки — 2 месяца
- **R-006:** Ограниченный бюджет (использовать бесплатные/opensource инструменты)

### 4.2. Функциональные требования

#### 4.2.1. Модуль "Аутентификация и профиль" (FR-001)

- **FR-001.1** Система должна позволять пользователю начать регистрацию по номеру телефона.
- **FR-001.2** Система должна отправлять SMS с 4-значным кодом подтверждения.
- **FR-001.3** Система должна верифицировать код и создавать/возвращать профиль клиента.
- **FR-001.4** Система должна позволять клиенту заполнить имя и email после регистрации.
- **FR-001.5** Система должна хранить JWT-токен для авторизации последующих запросов.

**User Story:** US-001 — Как Клиент, я хочу зарегистрироваться по номеру телефона, чтобы быстро создать аккаунт без запоминания паролей.

#### 4.2.2. Модуль "Расписание" (FR-002)

- **FR-002.1** Система должна отображать список доступных слотов тренировок на ближайшие 7 дней по умолчанию.
- **FR-002.2** Система должна позволять фильтровать слоты по дате, формату (новички/опытные) и инструктору.
- **FR-002.3** Система должна показывать для каждого слота: дату/время, формат, инструктора, количество свободных мест, статус (active/full/cancelled).
- **FR-002.4** Система должна отображать понятный Empty State, если на ближайшие 7 дней нет тренировок.
- **FR-002.5** Система должна показывать скелетоны (skeleton screens) во время загрузки данных.

**User Story:** US-002 — Как Клиент, я хочу видеть расписание тренировок на неделю вперёд, чтобы спланировать свои посещения.

#### 4.2.3. Модуль "Бронирование" (FR-003, FR-004, FR-005)

- **FR-003.1** Система должна позволять клиенту создать бронь на выбранный слот.
- **FR-003.2** Система должна позволять указать необходимость проката снаряжения (чекбокс).
- **FR-003.3** Система должна запрещать бронирование слота со статусом `full` или `cancelled_by_climbing_gym`.
- **FR-003.4** Система должна обрабатывать ошибку "Слот заполнен" (409 Conflict) и обновлять UI.
- **FR-003.5** Система должна отправлять email-подтверждение после успешного бронирования.

- **FR-005.1** Система должна позволять клиенту отменить бронь без ограничений, если до начала слота осталось более 2 часов.
- **FR-005.2** Система должна блокировать отмену и показывать объяснение, если до начала менее 2 часов.
- **FR-005.3** Система должна обновлять статус брони на `cancelled_by_client` и освобождать место.

**User Stories:**

- US-003 — Как Клиент, я хочу записаться на тренировку и выбрать прокат снаряжения, чтобы гарантированно получить место и экипировку.
- US-004 — Как Клиент, я хочу отменить свою запись минимум за 2 часа до начала, чтобы освободить место для других.

#### 4.2.4. Модуль "Оценки" (FR-006)

- **FR-006.1** Система должна позволять клиенту оставить оценку (1-5 звёзд) инструктору.
- **FR-006.2** Система должна позволять добавить необязательный текстовый комментарий (макс. 500 символов).
- **FR-006.3** Система должна разрешать оценку только в течение 24 часов после завершения тренировки.
- **FR-006.4** Система должна разрешать оценку только если клиент реально посетил тренировку (`booking.attended = true`).
- **FR-006.5** Система должна запрещать повторную оценку одной и той же брони (уникальность `booking_id`).

**User Story:** US-005 — Как Клиент, я хочу оценить инструктора после тренировки, чтобы поделиться своим опытом.

#### 4.2.5. Модуль "Лояльность" (FR-007)

- **FR-007.1** Система должна автоматически присваивать статус `is_loyal = true` клиенту, у которого количество завершённых броней (`status = completed`) достигло 10.
- **FR-007.2** Система должна отображать статус "Постоянный клиент" в профиле.
- **FR-007.3** Система должна применять скидку 10% для лояльных клиентов (отображать в UI, расчёт на стороне бэкенда).

**User Story:** US-006 — Как Постоянный клиент, я хочу получать скидку 10% после 10 посещений, чтобы чувствовать свою ценность для скалодрома.

#### 4.2.6. Модуль "Уведомления" (FR-008)

- **FR-008.1** Система должна отправлять email-уведомление клиенту за 2 часа до начала тренировки.
- **FR-008.2** Система должна отправлять email-уведомление клиенту в случае отмены слота скалодромом.
- **FR-008.3** Система должна отправлять email-подтверждение сразу после создания брони.
- **FR-008.4** Система должна хранить историю уведомлений в таблице Notification.

### 4.3. Нефункциональные требования

#### 4.3.1. Требования к интерфейсу (NFR-001)

- **NFR-001.1** Веб-интерфейс должен быть полностью адаптивным и корректно отображаться на мобильных устройствах (iOS Safari, Android Chrome).
- **NFR-001.2** Время отклика интерфейса при загрузке списка слотов не должно превышать 2 секунд при нормальных условиях сети.
- **NFR-001.3** Все интерактивные элементы должны иметь размер не менее 44x44 пикселей (рекомендации Apple HIG).
- **NFR-001.4** Контрастность текста и фона должна соответствовать WCAG 2.1 AA.

#### 4.3.2. Требования к надёжности (NFR-002)

- **NFR-002.1** Приложение должно корректно обрабатывать ошибки API бэкенда (отсутствие слотов, ошибка сети), отображая понятный Empty State или сообщение об ошибке.
- **NFR-002.2** Приложение должно работать в условиях нестабильного интернета (скалодром в ангаре) — показывать состояние загрузки, возможность повторить запрос.
- **NFR-002.3** Все критические операции (бронирование, отмена) должны иметь подтверждение от пользователя.

#### 4.3.3. Требования к безопасности

- **SEC-001** Пароли не используются (аутентификация по SMS).
- **SEC-002** JWT-токены должны храниться в httpOnly cookies или secure storage.
- **SEC-003** Все API-запросы, кроме публичных (GET /formats, /instructors, /slots), должны требовать авторизацию.
- **SEC-004** SMS-коды должны истекать через 5 минут.
- **SEC-005** Персональные данные (телефон, email) должны передаваться только по HTTPS.

#### 4.3.4. Требования к производительности

- **PERF-001** Время загрузки страницы расписания: <2 сек (3G), <1 сек (4G/WiFi).
- **PERF-002** Время отклика API при бронировании: <500 мс.
- **PERF-003** Поддержка до 500 одновременных пользователей (пиковая нагрузка в часы пик).

## 9. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Слот (TrainingSlot)** | Конкретное время проведения тренировки с указанием формата, инструктора и количества мест |
| **Бронь (Booking)** | Запись клиента на конкретный слот с указанием потребности в прокате снаряжения |
| **Формат (TrainingFormat)** | Тип тренировки: болдеринг (новички) или трассы с верёвкой (опытные) |
| **Лояльный клиент** | Клиент, посетивший 10 и более тренировок, получает скидку 10% |
| **Read-only сущность** | Данные, которые приложение только читает из внешнего бэкенда (TrainingFormat, Instructor, TrainingSlot) |
| **Mutable сущность** | Данные, которые приложение создаёт и изменяет (Client, Booking, Review, Notification) |
| **Empty State** | Экран-заглушка, показываемый при отсутствии данных (например, нет доступных тренировок) |
| **Skeleton Screen** | Анимированный placeholder, показываемый во время загрузки данных |
